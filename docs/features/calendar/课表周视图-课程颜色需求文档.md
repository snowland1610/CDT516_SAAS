# 课表周视图 - 课程颜色显示需求文档

> 用于指导课表周视图下「不同课程不同颜色」的预期效果与实现规范，便于排查当前颜色显示问题并统一交付标准。

---

## 一、目标与范围

### 1.1 目标

- **同一门课程**在周视图中的所有排课（不同班级、不同星期、不同时间段）使用**同一种颜色**。
- **不同课程**在周视图中尽量使用**不同颜色**，便于一眼区分。
- 学生端与教师端对**同一门课程**应显示**相同颜色**（一致体验）。

### 1.2 范围

- **视图**：仅课表**周视图**（学生端 `/student/calendar`、教师端 `/teacher/calendar`）。
- **元素**：周视图时间网格中的**课程事件块**（每个事件块对应一条 `section_schedule` 记录，通过 section → course 关联到课程）。
- 日视图、月视图若后续实现，应复用同一套「按课程着色」的规则与工具。

---

## 二、数据与语义

### 2.1 数据关系（当前库表）

- **section_schedule**：单条记录表示某班级(section)在某星期几、某时间段的排课。
- **section**：属于某门**课程(course)**，一个课程可有多个班级（多个 section）。
- **course**：课程实体，有唯一 `id`。

因此：

- 颜色应以**课程(course)**为维度，即以 **course_id** 为着色依据。
- 不应以 section_id、section_schedule.id 或「第几条排课」为依据，否则同一课程不同班级会变成不同颜色。

### 2.2 颜色键（Color Key）

| 项目     | 约定 |
|----------|------|
| 颜色键   | **course_id**（数字，来自 `section.course_id` 或关联的 `course.id`） |
| 取值保证 | 事件块渲染时必须能拿到有效的 course_id；若拿不到（数据异常），应有降级方案（见下文）。 |

---

## 三、理想交付效果（可验收）

### 3.1 同一课程 → 同一颜色

- 课程 A（如「SAAS设计」）的所有排课：
  - 周一 section1、周二 section2、……任意星期、任意 section，只要属于课程 A，事件块颜色**全部相同**。
- 课程 B（如「python编程」）的所有排课：同上，全部为另一种颜色，且与课程 A 不同（在调色板足够时）。

### 3.2 不同课程 → 不同颜色（在调色板容量内）

- 若当前周内出现的**不同课程数量** ≤ **调色板颜色数量**：每门课程对应一种颜色，且互不相同。
- 若课程数 > 调色板颜色数：允许多门课程共用一种颜色（按既定映射规则取模或顺序分配），但**同一课程**仍必须**同一颜色**。

### 3.3 跨端一致

- 学生端与教师端对**同一 course_id** 使用**同一颜色**。
- 实现上应使用同一套映射逻辑（如共用的 `getCourseColorClasses(courseId)`），且入参均为 course_id。

### 3.4 稳定性与确定性

- 同一 course_id 在任何周、任何用户、任何端，得到的颜色**一致**（不因「当前周内事件顺序」或「请求顺序」而变）。
- 即：颜色由 **course_id 唯一决定**，采用确定性算法（如 `f(course_id) → 颜色索引`），不依赖运行时顺序。

---

## 四、调色板与映射规则

### 4.1 调色板

- 使用一组数量有限的 Tailwind 类（如 8 种），覆盖：背景、边框、悬停态（如 `bg-*-200 border-*-400 hover:bg-*-300`）。
- 颜色之间应易于区分（蓝、绿、琥珀、紫、玫红、天蓝、翠绿、橙等），避免相邻色过于接近。

### 4.2 映射规则（建议）

- **输入**：`courseId: number`（有效且为整数）。
- **输出**：某一种 Tailwind 类组合（对应调色板中的一种）。
- **算法要求**：确定性。例如 `index = Math.abs(courseId) % paletteLength`，再取 `palette[index]`。同一 courseId 永远得到同一 index，进而同一颜色。
- **不建议**：按「当前周内事件出现顺序」或「先到先得」分配颜色，会导致同一课程在不同周或不同用户下颜色不一致。

### 4.3 边界与降级

| 情况           | 处理方式 |
|----------------|----------|
| course_id 正常 | 按上述映射规则取色。 |
| course_id 为 0 | 可视为有效（映射到调色板某一项），或单独定义为「中性色」（如灰色）。 |
| course_id 缺失/undefined/null | 使用统一的**降级样式**（如灰色背景 `bg-gray-200`），且控制台可打 warning，便于排查数据问题。 |
| 课程数 > 调色板数 | 允许多个 course_id 映射到同一颜色；不要求扩展调色板，但同一课程必须同色。 |

## 五、实现要点（供开发对照）

### 5.1 数据流

- 从 Supabase 拉取的 `section_schedule` 需带有关联：`section → course_id`（或 section.course.id）。
- 构建「周内事件」时，每个事件对象必须带有 **courseId**（即 course_id），且类型为 number。
- 渲染事件块时，**仅使用 courseId** 调用颜色工具（如 `getCourseColorClasses(ev.courseId)`），不要使用 section_id、schedule id 等。

### 5.2 样式应用

- 事件块根节点 className 中应包含：基础样式 + 由 `getCourseColorClasses(courseId)` 返回的类名。
- 确保没有其他更高优先级样式覆盖背景/边框（如内联 style 或其它 Tailwind 类覆盖了背景）。
- 若使用 Tailwind，需保证对应颜色类未被 purge 掉（通常保留在源码中即可）。

### 5.3 共用工具

- 颜色工具（如 `lib/calendarColors.ts`）应只接受 **courseId**，返回同一套类名字符串。
- 学生端与教师端都调用同一工具、同一入参语义（course_id），不各自实现一套映射。

## 六、验收标准（Checklist）

实现或修复后，可用以下项自检：

- [ ] **同一课程多节次**：同一门课在不同星期、不同班级的多个事件块，颜色完全一致。
- [ ] **不同课程**：至少两门课在同一周出现时，显示为两种不同颜色。
- [ ] **学生/教师一致**：同一门课在学生端与教师端为同一颜色。
- [ ] **换周不变**：同一门课在不同周之间颜色不变。
- [ ] **无 course 时**：若某条排课因数据问题没有 course_id，使用降级色且控制台有提示，不报错。
- [ ] **可读性**：事件块内文字（课程名、教室、时间等）在背景色上对比度足够，可读。

## 七、与现有实现的对应关系

| 需求点           | 当前实现（参考） |
|------------------|------------------|
| 颜色键           | `ev.courseId` 来自 `row.section.course_id`，需确认查询与类型正确。 |
| 颜色工具         | `lib/calendarColors.ts` 的 `getCourseColorClasses(courseId)`，按 courseId 取模。 |
| 学生端渲染       | `app/student/calendar/page.tsx` 中事件块使用 `getCourseColorClasses(ev.courseId)`。 |
| 教师端渲染       | `app/teacher/calendar/page.tsx` 同上。 |
| 调色板           | `COURSE_COLOR_CLASSES` 数组，约 8 种。 |

排查「颜色显示不正确」时，可依次检查：

1. 接口返回的 `section.course_id` 是否存在且为数字。
2. `WeekEvent` 的 `courseId` 是否始终等于该 course_id。
3. `getCourseColorClasses(ev.courseId)` 是否被正确拼接到 className，且无被覆盖。
4. 是否存在多处事件块渲染逻辑，某处误用了 section_id 或 schedule id。
