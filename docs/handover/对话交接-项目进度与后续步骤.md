# 对话交接：项目进度与后续步骤

> 本文档供「新对话」快速理解项目目标、已完成内容、技术约定与剩余阶段，便于无缝继续开发。

---

## 一、项目是什么

- **产品**：高校教务协作 SAAS（演示用），打通**教务处、教师、学生**三端信息，核心功能包括**小组组队**与**小组任务管理**（任务发布→组队→提交→批改→成绩）。
- **技术**：Next.js 14（App Router）+ Tailwind CSS + Supabase Client，**直接连接 Supabase 云端**（无 Docker/本地库）；登录方式为**选身份 + 选用户**，无密码。
- **数据**：表结构与演示数据在项目内用 `.sql` 维护；**你（或用户）**把 SQL 复制到 Supabase 网页的 SQL Editor 执行，表在云端生效；前端只连云端 Supabase（`.env.local` 中的 URL + anon key）。

## 三、技术约定（勿改）

- **环境变量**：`NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY` 在 `.env.local`，勿提交。
- **Supabase 客户端**：`lib/supabase.ts` 的 `createClient()`（来自 `@supabase/ssr` 的 createBrowserClient），全站用这一份连云端。
- **当前用户**：`context/AuthContext.tsx`，`CurrentUser` 含 `id, role, display_name, username?, profile_id?`；`role` 为 `admin \| teacher \| student`；选用户后 `selectUser(u)`，登出 `logout()` 并跳 `/`。
- **表名**：库里有表 `"user"`（保留字），Supabase 查询用 `.from('user')`；teacher/student 联表时外键展开的键可能是 `user_id`（即 `user_id: { id, display_name, username }`）。
- **路由守卫**：`/admin/*`、`/teacher/*`、`/student/*` 的 `layout.tsx` 内用 `useAuth()` 校验 `user?.role`，不符则 `router.replace('/')` 或 `router.replace('/select-user?role=xxx')`。

## 五、已完成阶段（0～2）与验证情况

- **阶段 0**：Next.js + Tailwind + Supabase client + AuthContext + `.env.local`；可 `npm run dev`，前端连云端。
- **阶段 1**：主页面三端入口、选用户页（从 Supabase 拉 admin/教师/学生列表）、选后写入 Context 并跳对应工作台；三端占位工作台页。
- **阶段 2**：Header、三端 Sidebar、三端 layout（路由守卫 + 顶栏+侧栏+主区）；侧栏对应占位页（课程、公告、统计等）；工作台页简化，去掉重复顶栏。
- **已验证**：登录流程（选身份→选用户→进工作台）和三端框架（顶栏、侧栏、守卫）可用。

## 七、新对话接到「继续开发」时建议动作

1. 打开并扫一眼 **`docs/planning/程序结构与页面生成顺序规划.md`** 和 **`docs/planning/页面设计需求文档.md`**，确认阶段 3～8 的页面与路由。
2. 从 **阶段 3** 开始：实现教务处工作台（可先简单统计卡片 + 近期公告占位）、课程目录页（从 Supabase 查 course 联表 major、teacher，并统计 section/enrollment）、课程详情页（班级列表、section_schedule 课表、选课学生）。
3. 做阶段 4、5 时同样对照「页面设计需求文档」中的教师端/学生端页面与数据来源。
4. 进入阶段 6 前：在项目内新增 **announcement 表** 的建表 SQL，并在文档中说明「请用户将本段 SQL 复制到 Supabase SQL Editor 执行」；再实现公告的增删改查与三端展示。
5. 进入阶段 7 前：同上，提供 **task、group、group_member、submission** 的建表 SQL，用户执行后再实现任务发布、组队、提交、批改、成绩查看。
6. 所有新页面保持：**只连 Supabase 云端**（不写死 URL/key）、用 **AuthContext** 取当前用户与 profile_id、遵守现有 **layout（Header + Sidebar）** 与路由守卫。

## 八、Supabase 与 SQL 的约定

- 项目内已有 **`supabase/schema-and-seed.sql`**（建表+演示数据），用户已在云端执行过，故当前表与数据存在。
- 后续新增表（如 announcement、task、group、group_member、submission）时：在项目内**新增或修改 .sql 文件**，在文档或注释中写明「请将以下 SQL 复制到 Supabase 项目 → SQL Editor 中执行」；不在对话中代替用户执行云端操作。
