# 学生端组队邀请与全班名单 — 修改计划

> 在现有组队大厅基础上：展示本 section 全班同学组队情况；组长可邀请未组队同学；被邀请人可在一栏「收到的邀请」中接受/拒绝，并展示近期已处理记录。供后续编程实施使用。

---

## 一、需求与约定摘要（已确认）

| 项 | 结论 |
|----|------|
| Section 维度 | 学生同一门课下最多只属于一个 section；组队大厅只展示**当前所选课程下自己所在的 section** 的同学情况，无需 section 选择器 |
| 邀请存储 | 新增表管理邀请；同一「小组 → 被邀请人」只允许一条未处理邀请，重复邀请视为更新 |
| 自动失效 | 被邀请人已加入该 section 下任意小组时，将该 section 下对其的未处理邀请自动置为拒绝 |
| 收到的邀请展示 | 显示近期已接受/已拒绝记录，仅展示状态（如「已接受」「已拒绝」） |
| 邀请入口 | 对本功能逻辑最优：在「未组队同学」列表中，每行未组队同学后显示「邀请」按钮（仅组长可见）；不能跨 section，仅限当前 section、未组队同学 |
| 接受约束 | 接受时校验：小组未满（如 ≤6 人）；该同学在当前 section 下未加入其他小组。不满足时提示「小组已满」或「您已在其他小组」 |
| 收到的邀请位置 | 方案 A：在组队大厅页内，**选中课程后**，在「组队内容区」**上方**增加一块「收到的邀请」 |

## 二、数据表设计

### 2.1 新增表：group_invitation

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGSERIAL PRIMARY KEY | 主键 |
| `group_id` | BIGINT NOT NULL REFERENCES "group"(id) ON DELETE CASCADE | 被邀请加入的小组 |
| `inviter_id` | BIGINT NOT NULL REFERENCES student(id) ON DELETE CASCADE | 邀请人（组长） |
| `invitee_id` | BIGINT NOT NULL REFERENCES student(id) ON DELETE CASCADE | 被邀请人 |
| `status` | VARCHAR(20) NOT NULL DEFAULT 'pending' | `pending` 待处理 / `accepted` 已接受 / `rejected` 已拒绝 |
| `created_at` | TIMESTAMPTZ DEFAULT now() | 创建时间 |
| `updated_at` | TIMESTAMPTZ DEFAULT now() | 更新时间（接受/拒绝时更新） |

- **约束**：同一 (group_id, invitee_id) 仅允许一条 `status = 'pending'` 的记录，用**部分唯一索引**实现：  
  `CREATE UNIQUE INDEX ... ON group_invitation(group_id, invitee_id) WHERE status = 'pending';`
- **重复邀请**：若已存在该 (group_id, invitee_id) 的 pending 记录，则更新 `updated_at`（或不做操作）；若之前为 accepted/rejected，则可插入新一条 pending 或更新为 pending（按产品约定，本计划采用：仅允许一条 pending，历史记录保留为 accepted/rejected，不覆盖）。
- **索引**：`invitee_id`（查收到的邀请）、`group_id`、`(invitee_id, status)` 便于列表查询。

### 2.2 建表 SQL 存放

- 文件：`supabase/migrations/YYYYMMDD_add_group_invitation_table.sql`
- 说明：在文件头部注明「请将本 SQL 复制到 Supabase 项目 → SQL Editor 中执行」。

## 三、业务逻辑补充

### 3.1 自动拒绝逻辑

- **时机**：被邀请人通过任意方式加入该 section 下某小组时（包括：点击「加入」、接受其他邀请）。
- **实现**：在应用层，每当向 `group_member` 插入一条记录后，查询该 `group` 的 `section_id`，将该 section 下所有「邀请该学生且 status = pending」的邀请更新为 `status = 'rejected'`，即：  
  `UPDATE group_invitation SET status = 'rejected', updated_at = now() WHERE invitee_id = ? AND status = 'pending' AND group_id IN (SELECT id FROM "group" WHERE section_id = ?)`。

### 3.2 邀请按钮显示条件

- 当前用户在该 section 下是**某个小组的组长**（`group.leader_id = 当前学生`）。
- 对象同学在该 section 下**未加入任何小组**（不在任何 `group_member` 中，且该 group 的 section_id = 当前 section）。
- 仅对**当前 section** 的未组队同学显示邀请，不跨 section。

### 3.3 接受邀请时的校验（应用层）

1. 该小组当前人数 < 小组人数上限（如 6）。
2. 被邀请人（当前用户）在该 section 下未加入任何其他小组。
3. 不满足 1 则提示「小组已满」；不满足 2 则提示「您已在其他小组」，并将该条邀请置为 `rejected`（可选）。

### 3.4 重复邀请

- 若已存在 (group_id, invitee_id, status = 'pending')，组长再次点击「邀请」时：更新该记录的 `updated_at` 即可（视为「再次提醒」），不插入新记录。
- 若之前为 accepted/rejected，是否允许「再次邀请」由产品决定；本计划约定：允许，即插入一条新的 pending（此时旧记录已非 pending，唯一约束不冲突）。

## 四、页面与交互（组队大厅页）

### 4.1 本 section 全班同学名单

- **位置**：在组队大厅页选中课程后，当前学生在该课程下仅有**一个 section**（业务约定）；在「组队内容区」内，与「我的小组 / 创建小组 / 可加入的小组」并列或在其上，增加一块「本班同学组队情况」。
- **数据**：  
  - 该 section 下所有选课学生：`enrollment` 中 `section_id = 当前 section` 的 `student_id`，关联 `student` 取姓名、学号。  
  - 每人组队状态：通过 `group_member` + `group`（`section_id = 当前 section`）得到「该学生在哪个小组」；若不在任何小组则为「未组队」。
- **展示**：表格或列表，列：姓名、学号、组队状态（小组名称或「未组队」）。对**未组队**且**当前用户是该 section 下某小组组长**的行，显示「邀请」按钮。
- **邀请操作**：点击「邀请」→ 若已有 (group_id, invitee_id) 且 status = pending，则仅刷新或更新 updated_at；否则 `INSERT INTO group_invitation (group_id, inviter_id, invitee_id, status) VALUES (?, 当前用户, ?, 'pending')`（需保证 inviter 为组长，应用层校验）。

### 4.2 收到的邀请

- **位置**：在组队大厅页内，**选中课程后**，在「组队内容区」**上方**增加一块「收到的邀请」。
- **数据**：`group_invitation` 中 `invitee_id = 当前用户`，且邀请所在小组的 `section` 属于当前所选课程（可选：仅显示与当前所选课程相关的邀请，便于聚焦；或显示全部待处理 + 近期已处理）。按 `created_at` 倒序。
- **展示**：列表，每项：小组名、邀请人（组长）姓名、课程/班级名、时间、状态（待处理 / 已接受 / 已拒绝）。  
  - **待处理**：操作「接受」「拒绝」。  
  - **已接受 / 已拒绝**：仅展示状态文案，无操作。
- **接受**：校验小组未满、本人当前 section 未加入其他组；通过则插入 `group_member`，更新邀请 `status = 'accepted'`，并执行「自动拒绝」该 section 下对该用户的其他 pending 邀请；不通过则提示且不修改邀请状态（或置为 rejected）。  
- **拒绝**：更新该条邀请 `status = 'rejected'`。

## 六、与现有代码的衔接

- **组队大厅页**：当前已有课程选择、按 section 展示「我的小组 / 创建小组 / 可加入的小组」。学生在该课程下仅一个 section，因此 `mySections` 取第一个（或唯一一个）即可，用于「本班同学」与「邀请」范围。
- **小组人数上限**：与现有一致（如 6），从 `group` 所在 section 或任务配置读取；本计划可写死 6，与现有 `MAX_GROUP` 一致。
- **样式**：沿用现有设计基线（`rounded-card`、`shadow-card`、表格/列表风格）。

## 七、验收要点

- 组队大厅选中课程后，能看到本 section 全班同学名单及每人组队状态（小组名或「未组队」）。
- 组长在未组队同学行可点击「邀请」；同一 (小组, 被邀请人) 仅一条 pending，重复邀请视为更新或忽略。
- 被邀请人在「收到的邀请」中看到待处理邀请，可接受或拒绝；接受时校验小组未满、本人未加入其他组，通过则加入小组并自动拒绝该 section 下其余 pending 邀请。
- 被邀请人通过「加入」或「接受邀请」加入某小组后，该 section 下对其的其它未处理邀请自动变为已拒绝。
- 「收到的邀请」展示近期已接受/已拒绝记录，仅展示状态。
