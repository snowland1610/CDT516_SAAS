# 程序结构与页面生成顺序规划

> 基于《页面设计需求文档》+「直接连 Supabase 云端」方案；约定整体结构、生成顺序与每步产出，便于按阶段交付。

---

## 一、技术栈与数据连接约定

| 项目 | 选型 / 约定 |
|------|----------------|
| 框架 | Next.js 14（App Router） |
| 样式 | Tailwind CSS |
| 组件 | shadcn/ui |
| 数据 | Supabase Client（@supabase/supabase-js），**直连云端项目** |
| 环境变量 | `.env.local` 中 `NEXT_PUBLIC_SUPABASE_URL`、`NEXT_PUBLIC_SUPABASE_ANON_KEY`（你已在网页创建的项目的 URL 与 anon key） |
| 当前用户 | React Context（选身份 + 选用户后写入，无密码） |
| SQL 与库 | 我维护项目内 `.sql` 文件；你复制到 Supabase SQL Editor 执行，表结构/数据在云端生效；前端只连云端 Supabase |

- **路由守卫**：`/admin/*`、`/teacher/*`、`/student/*` 的 layout 内校验当前用户角色与路由是否匹配，未登录或角色不符则重定向到 `/` 或 `/select-user`。

## 三、页面生成顺序与阶段划分

按「先跑通登录与三端框架，再按角色补齐列表/详情，最后公告与任务/组队/提交」的顺序生成。

---

### 阶段 0：项目初始化与基础设施（必须先完成）

**目标**：本地能跑 Next.js，前端能连上 Supabase 云端，且当前用户状态可用。

| 步骤 | 内容 | 产出 | 你需要做的 |
|------|------|------|------------|
| 0.1 | 创建 Next.js 14 项目（App Router）、安装 Tailwind、shadcn/ui、@supabase/supabase-js | 可跑 `npm run dev`、基础 UI 组件 | 无（我生成后你执行 `npm install`、`npm run dev`） |
| 0.2 | 配置 `.env.local` 示例、`lib/supabase.ts`（createBrowserClient） | 前端用 env 连 Supabase | 把示例里的占位换成你云端项目的 **URL** 和 **anon key**，并确保 Supabase 已执行过建表+seed |
| 0.3 | 实现 `context/AuthContext`（当前 user、role、选用户、登出） | 全局可读当前用户、选完用户可写状态并跳转 | 无 |
| 0.4 | 根布局 `app/layout.tsx` 包一层 `AuthProvider` | 全站可用 Context | 无 |

**完成标志**：项目能启动，Supabase 连接与 Auth 上下文就绪（尚未做页面时可用临时首页或 console 自测）。

---

### 阶段 1：主入口与选用户（登录流程）

**目标**：主页面三个入口 → 选用户 → 按角色跳转工作台（工作台可为占位）。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 1.1 | 主页面 `app/page.tsx`：标题 + 三个入口卡片（教务处 / 教师 / 学生），点击跳转 `/select-user?role=xxx` | 首页 | 无 |
| 1.2 | 选用户页 `app/select-user/page.tsx`：根据 `role` 从 Supabase 查 `user`（+ teacher/student 表），列表展示；点击某项写入 Context 并跳转 `/admin`、`/teacher`、`/student` | 选用户 + 进入对应端 | `user`、`teacher`、`student` 表（已有） |

**完成标志**：选「教务处管理员」进 `/admin`，选某教师进 `/teacher`，选某学生进 `/student`（此时各端可为空白或简单占位）。

---

### 阶段 2：公共布局与路由守卫

**目标**：三端各有统一顶栏 + 侧栏，未登录或角色错时重定向。

| 步骤 | 内容 | 产出 | 依赖 |
|------|------|------|------|
| 2.1 | `components/layout/Header.tsx`：应用名、当前角色与用户名、登出（清 Context 并跳 `/`） | 顶栏 | AuthContext |
| 2.2 | `app/admin/layout.tsx`：校验 role=admin，否则重定向；左侧 AdminSidebar（工作台、课程目录、公告）、右侧 children；顶栏复用 Header | 教务处布局 | AuthContext |
| 2.3 | `app/teacher/layout.tsx`、`app/student/layout.tsx` 同理（侧栏项按需求文档） | 教师/学生布局 | AuthContext |

**完成标志**：进入 `/admin` 等需已选对应用户，否则回到首页或选用户页；侧栏可点，对应页面可后续逐步实现。

---

### 阶段 3：教务处端（课程与工作台）

**目标**：教务处工作台、课程目录、课程详情，全部用现有表（无需公告表）。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 3.1 | `app/admin/page.tsx`：工作台：统计卡片（课程总数等）、近期公告占位「暂无公告」、快捷入口（课程目录、公告） | 教务处工作台 | `course` 计数（可选） |
| 3.2 | `app/admin/courses/page.tsx`：课程目录列表（course + major + teacher，统计 section/enrollment） | 课程目录 | course, major, teacher, section, enrollment |
| 3.3 | `app/admin/courses/[id]/page.tsx`：课程详情（基本信息、班级列表、每班选课人数、课表 section_schedule、可选学生名单） | 课程详情 | course, section, section_schedule, enrollment, student |

**完成标志**：教务处可查看全部课程及课程下班级、课表、选课人数（及学生名单若实现）。

---

### 阶段 4：教师端（课程、课表、学生名单）

**目标**：教师工作台、我的课程、课程详情（班级与课表、学生名单）；任务相关仅占位或简单列表（若尚无 task 表）。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 4.1 | `app/teacher/page.tsx`：工作台（我的课程数、公告占位、快捷入口「我的课程」） | 教师工作台 | course 按 teacher_id 计数 |
| 4.2 | `app/teacher/courses/page.tsx`：我的课程列表（当前教师负责的 course，班级数、选课人数） | 我的课程 | course, section, enrollment |
| 4.3 | `app/teacher/courses/[id]/page.tsx`：课程详情（基本信息、Tab：班级与课表、学生名单、任务列表占位）、按钮「发布任务」跳转占位 | 课程详情 | course, section, section_schedule, enrollment, student |

**完成标志**：教师能看自己负责的课程、每门课的班级与课表、选课学生名单；任务列表可为「暂无任务」或占位。

---

### 阶段 5：学生端（我的课程、课表）

**目标**：学生工作台、我的课程、课程详情（课表、任务列表占位）；任务详情与组队/提交留待有 task 表后再做。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 5.1 | `app/student/page.tsx`：工作台（我的课程列表、公告占位、待办占位） | 学生工作台 | enrollment → section → course |
| 5.2 | `app/student/courses/page.tsx`：我的课程（当前学生选中的 course + section，教师名） | 我的课程 | enrollment, section, course, teacher |
| 5.3 | `app/student/courses/[id]/page.tsx`：课程详情（课程名、教师、所在 section、该 section 课表 section_schedule、任务列表占位） | 课程详情 | section_schedule, task 可选 |

**完成标志**：学生能看自己选的课、每门课所在班级与课表；任务区可为「暂无任务」或占位。

---

### 阶段 6：公告（需公告表）

**目标**：教务处发布/编辑/列表公告，教师与学生只读列表与详情。**前提**：你已在 Supabase 执行我提供的 `announcement` 表 SQL。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 6.0 | 我提供 `announcement` 表建表 SQL；你复制到 Supabase SQL Editor 执行 | 云端有 announcement 表 | — |
| 6.1 | `app/admin/announcements/page.tsx`：列表 + 按钮「发布公告」；`new`、`[id]/edit` 表单页 | 公告管理 | announcement |
| 6.2 | `app/teacher/announcements/page.tsx`、`app/student/announcements/page.tsx`：只读列表 + 详情 | 教师/学生看公告 | announcement |
| 6.3 | 工作台「近期公告」改为真实数据（admin/teacher/student 三端） | 工作台展示公告 | announcement |

**完成标志**：教务处能发公告，教师和学生能在侧栏进公告列表并看详情；工作台显示近期公告。

---

### 阶段 7：任务、小组、提交与批改（需 task / group / submission 表）

**目标**：教师发布任务 → 学生看到任务、组队、提交 → 教师批改并发布成绩。**前提**：你已在 Supabase 执行我提供的 task、group、group_member、submission 表 SQL。

| 步骤 | 内容 | 产出 | 依赖数据 |
|------|------|------|----------|
| 7.0 | 我提供 task、group、group_member、submission 建表 SQL；你复制到 Supabase 执行 | 云端有上述表 | — |
| 7.1 | 教师：发布任务页、课程详情中任务列表与任务详情（提交列表）、批改页 | 教师端任务与批改 | task, group, group_member, submission |
| 7.2 | 学生：课程详情中任务列表、任务详情（组队、提交、成绩展示） | 学生端任务与提交 | task, group, group_member, submission |
| 7.3 | 工作台待办/统计与任务、提交状态联动（可选） | 工作台数据完善 | 同上 |

**完成标志**：演示流程可完整跑通：教师发任务 → 学生组队并提交 → 教师批改发布成绩 → 学生查成绩。

---

### 阶段 8：统计与收尾（可选）

| 步骤 | 内容 | 产出 |
|------|------|------|
| 8.1 | `app/admin/stats/page.tsx`：按课程汇总任务数、提交率、平均分等 | 教务处统计页 |
| 8.2 | 样式与响应式微调、错误态与空态统一 | 整体体验 |

## 四、生成顺序与依赖关系简图

```
阶段 0（初始化 + Supabase 连接 + Auth）
    ↓
阶段 1（主页面 + 选用户）
    ↓
阶段 2（三端 layout + 路由守卫）
    ↓
阶段 3（教务处：工作台 + 课程目录 + 课程详情）
    ↓
阶段 4（教师：工作台 + 我的课程 + 课程详情）
    ↓
阶段 5（学生：工作台 + 我的课程 + 课程详情）
    ↓
阶段 6（公告：需你先执行 announcement 表 SQL）
    ↓
阶段 7（任务/组队/提交/批改：需你先执行 task/group/submission 表 SQL）
    ↓
阶段 8（统计与收尾，可选）
```

- **阶段 0～5**：只依赖现有 Supabase 表（major, user, teacher, student, course, section, section_schedule, enrollment），无需你额外执行 SQL（只要你云端已执行过之前的 schema-and-seed）。
- **阶段 6**：需先执行我提供的 `announcement` 表 SQL。
- **阶段 7**：需先执行我提供的 task、group、group_member、submission 表 SQL。

## 五、你需要在各阶段配合的事

| 阶段 | 你需要做的 |
|------|------------|
| 0 | 在项目根目录创建 `.env.local`，填入 Supabase 云端的 **Project URL** 和 **anon key**；若云端尚未建表，把之前我给的 `schema-and-seed.sql` 在 Supabase SQL Editor 执行一次。 |
| 1～5 | 无需再执行 SQL；若某页报错，把报错信息发给我即可。 |
| 6 | 我给出 `announcement` 表 SQL 后，你复制到 Supabase SQL Editor 执行。 |
| 7 | 我给出 task、group、group_member、submission 表 SQL 后，你复制到 Supabase SQL Editor 执行。 |

## 六、与《页面设计需求文档》的对应关系

- 所有路由、页面结构、区块与数据来源均按《页面设计需求文档》执行。
- 本文档仅补充：**整体目录结构**、**按阶段生成页面的顺序**、**每阶段依赖的 Supabase 表**、**你何时需要执行哪份 SQL**，便于你按步骤配合、我按阶段生成页面与 SQL。

后续按「阶段 0 → 1 → 2 → …」依次生成即可；你随时可以说「从阶段 0 开始」或「做到阶段 3 了，继续阶段 4」，我会按当前阶段产出对应代码与说明。
