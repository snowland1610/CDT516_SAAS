## 一、交付范围总览

| 类别 | 交付内容 |
|------|----------|
| **数据库** | `task`、`submission` 表建表 SQL（+ 可选 seed），用户复制到 Supabase SQL Editor 执行；`group`、`group_member` 已存在，不重复建表 |
| **教师端** | 发布任务页；课程详情内任务列表（真实数据）与任务详情（提交列表）；批改页（分数、评语、发布成绩） |
| **学生端** | 课程详情内任务列表（真实数据）；任务详情（组队、提交、成绩展示） |
| **衔接** | 教师课程详情「任务列表」Tab、学生课程详情「任务列表」区块由占位改为真实数据；「发布任务」按钮由占位跳转改为真实发布页 |

**不交付**：附件上传到 Supabase Storage（本阶段提交内容仅支持文本/链接，`attachment_url` 字段预留）；工作台待办/统计与任务联动（阶段 7.3 可选）；教务处统计页（阶段 8）。

**完成标志**：教师发任务 → 学生看任务 → 学生组队（需组队时）并提交 → 教师批改并发布成绩 → 学生查成绩，演示流程可完整跑通。

## 二、数据表设计

### 2.1 已有表（仅引用，不新建）

- **group**：`id`, `section_id`, `name`, `leader_id`, `created_at`, `updated_at`（见 `supabase/migrations/20250228100000_add_group_tables.sql`）
- **group_member**：`id`, `group_id`, `student_id`, `joined_at`, ...（同上）

任务以**课程（course）**为单位发布，同一课程下所有 section 共用同一批任务；学生按** section** 组队（现有逻辑），提交时：若任务允许组队则以**组**为单位提交，否则以**个人**为单位提交。

### 2.2 task 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGSERIAL PRIMARY KEY | 主键 |
| `course_id` | BIGINT NOT NULL REFERENCES course(id) ON DELETE CASCADE | 所属课程 |
| `title` | VARCHAR(256) NOT NULL | 任务标题 |
| `description` | TEXT | 任务说明 |
| `due_at` | TIMESTAMPTZ NOT NULL | 提交截止时间 |
| `allow_group` | BOOLEAN NOT NULL DEFAULT true | 是否允许组队 |
| `group_min` | INT | 小组最少人数（允许组队时有效，如 4） |
| `group_max` | INT | 小组最多人数（允许组队时有效，如 6） |
| `group_deadline` | TIMESTAMPTZ | 组队截止时间（允许组队时可选；NULL 表示与 due_at 一致或不做限制） |
| `created_at` | TIMESTAMPTZ DEFAULT now() | 创建时间 |
| `updated_at` | TIMESTAMPTZ DEFAULT now() | 更新时间 |

- **约束**：`allow_group = false` 时，提交以个人为单位；`allow_group = true` 时，学生需先在该任务所属课程的某个 section 下组队，以组为单位提交。
- **索引**：`course_id` 用于按课程查任务列表。

### 2.3 submission 表

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGSERIAL PRIMARY KEY | 主键 |
| `task_id` | BIGINT NOT NULL REFERENCES task(id) ON DELETE CASCADE | 所属任务 |
| `group_id` | BIGINT REFERENCES "group"(id) ON DELETE CASCADE | 小组提交时必填；个人提交时为 NULL |
| `student_id` | BIGINT REFERENCES student(id) ON DELETE CASCADE | 个人提交时必填；小组提交时为 NULL（小组交一份即可） |
| `content` | TEXT | 提交内容（文本或链接） |
| `attachment_url` | VARCHAR(512) | 附件 URL（本阶段可留空，预留） |
| `score` | NUMERIC(5,2) | 成绩（批改后填写；NULL 表示未批改） |
| `feedback` | TEXT | 教师评语 |
| `status` | VARCHAR(20) NOT NULL DEFAULT 'submitted' | `submitted` 已提交 / `graded` 已批改 |
| `submitted_at` | TIMESTAMPTZ DEFAULT now() | 提交时间 |
| `graded_at` | TIMESTAMPTZ | 批改时间 |
| `created_at` | TIMESTAMPTZ DEFAULT now() | 创建时间 |
| `updated_at` | TIMESTAMPTZ DEFAULT now() | 更新时间 |

- **约束**：同一任务下，(group_id, student_id) 至少有一个非空；若 `group_id` 非空则为小组提交（该组交一份），若 `group_id` 为空则 `student_id` 必填为个人提交。每组/每人每任务只能提交一次：使用**部分唯一索引**实现——`UNIQUE (task_id, group_id) WHERE group_id IS NOT NULL` 与 `UNIQUE (task_id, student_id) WHERE student_id IS NOT NULL`（PostgreSQL）；应用层插入前也需校验避免重复提交。
- **索引**：`task_id`、`group_id`、`student_id` 便于查询与列表展示。

### 2.4 建表 SQL 存放位置与说明

- 文件：`supabase/migrations/YYYYMMDD_add_task_submission_tables.sql`（具体日期由实现时定）。
- 文档说明：在文件头部注释写明「请将本 SQL 复制到 Supabase 项目 → SQL Editor 中执行」。
- 可选：写入 1～2 条 task seed、0～1 条 submission seed 便于联调；或完全通过界面产生数据。

## 三、路由与页面清单

| 路径 | 角色 | 页面用途 |
|------|------|----------|
| `/teacher/courses/[id]` | 教师 | 课程详情（已有）；「任务列表」Tab 改为从 task 表拉取，每项可点进任务详情 |
| `/teacher/courses/[id]/tasks/new` | 教师 | 发布任务（表单 + 提交） |

## 四、页面与逻辑说明

### 4.1 教师端：发布任务 `/teacher/courses/[id]/tasks/new`

- **布局**：沿用 teacher layout；面包屑或标题「发布任务」+ 返回课程详情。
- **数据**：根据 `params.id` 校验当前用户为该课程教师（与课程详情页一致）。
- **表单**：
  - 任务标题：单行，必填。
  - 任务说明：多行文本，选填。
  - 提交截止时间：日期时间，必填。
  - 是否允许组队：单选/开关，默认 true。
  - 小组人数范围：当「允许组队」时显示，最小人数、最大人数（如 4～6），必填。
  - 组队截止时间：当「允许组队」时显示，可选；可不填表示与提交截止一致或不做限制。
- **提交**：`supabase.from('task').insert({ course_id, title, description, due_at, allow_group, group_min, group_max, group_deadline })`；成功后跳转 `/teacher/courses/[id]` 并定位到任务列表 Tab（或 toast 后留在列表）。
- **返回**：「返回课程详情」→ `/teacher/courses/[id]`。

### 4.2 教师端：课程详情内任务列表（替换占位）

- **位置**：`app/teacher/courses/[id]/page.tsx`，当前 `activeTab === 'tasks'` 的占位内容。
- **数据**：`supabase.from('task').select('*').eq('course_id', courseId).order('due_at', { ascending: true })`（或按 created_at 倒序）。
- **列表**：每行显示：任务标题、截止时间、是否组队、提交数/总组数（或总人数，见下）、操作「查看」。
- **提交数/总组数**：若任务 `allow_group = true`，总数为该课程下所有 section 的 group 数（需按 section → group 统计）；若 `allow_group = false`，总数为该课程选课学生数。已提交数来自 submission（按 task_id 统计，组提交算 1 条，个人提交每人 1 条）。本阶段可简化为：只显示「已提交数」，总数为「该课程下 section 对应的 group 数量之和 + 未组队人数」或仅显示「已提交 X 份」。
- **操作**：「查看」→ 跳转 `/teacher/courses/[courseId]/tasks/[taskId]`。
- **空态**：无任务时展示「暂无任务」+ 引导「发布任务」。

### 4.3 教师端：任务详情（提交列表）`/teacher/courses/[courseId]/tasks/[taskId]`

- **布局**：标题为任务名称 + 返回课程详情；展示任务信息（标题、说明、截止时间、组队规则）。
- **数据**：根据 `taskId` 拉取 task 单条；拉取该 task 下所有 submission（含 group、student 等关联，用于展示小组名/成员或学号）。
- **提交列表**：表格或卡片：小组名（若组提交）/ 学号姓名（若个人提交）、提交时间、提交内容（content 摘要）、状态（已提交/已批改）；每行操作「批改」→ `/teacher/courses/[courseId]/tasks/[taskId]/grade/[submissionId]`。
- **统计**：提交率 = 已提交数 / 应提交总数（应提交总数：组队任务 = 该课程各 section 的 group 数之和，个人任务 = 该课程选课学生数）；可显示「未提交小组/人数」。
- **权限**：校验该 task 的 course 的 teacher_id 为当前用户。

### 4.4 教师端：批改页 `/teacher/courses/[courseId]/tasks/[taskId]/grade/[submissionId]`

- **布局**：标题「批改：任务名」+ 小组/提交标识；返回任务详情。
- **数据**：拉取 submission 单条（含 task、group、student 等）；拉取 task 用于标题与说明。
- **展示**：提交内容（content）、附件链接（若有）；若组提交显示小组名与成员列表。
- **表单**：分数（数字输入）、评语（多行文本）；按钮「保存」与「发布成绩」。
- **保存**：仅更新 submission 的 score、feedback、updated_at（不改变 status），便于暂存。
- **发布成绩**：更新 submission 的 score、feedback、status = 'graded'、graded_at = now()、updated_at；成功后跳转任务详情或列表。
- **权限**：校验 submission 所属 task 的 course 的 teacher_id 为当前用户。

### 4.5 学生端：课程详情内任务列表（替换占位）

- **位置**：`app/student/courses/[id]/page.tsx`，当前「任务列表」占位区块。
- **数据**：`supabase.from('task').select('*').eq('course_id', courseId).order('due_at', { ascending: true })`。
- **列表**：每项显示：任务标题、截止时间、是否需组队、本人/本组提交状态。状态逻辑：若任务 allow_group，则查当前学生在当前课程某 section 下的 group，再查 submission 中 task_id + group_id；若不允许组队，查 submission 中 task_id + student_id。状态：未提交、已提交、已批改（status = 'graded'）。
- **操作**：点击任务 → 跳转 `/student/courses/[courseId]/tasks/[taskId]`。
- **空态**：「暂无任务」。

### 4.6 学生端：任务详情 `/student/courses/[courseId]/tasks/[taskId]`

- **布局**：标题为任务名称 + 返回课程详情；展示任务说明（标题、说明、截止时间、组队规则：人数范围、组队截止时间）。
- **我的小组**（当任务 allow_group 时）：
  - 若已组队：显示小组名、成员列表、组长；操作「提交作业」（若未提交）、或展示「已提交」+ 提交时间；若已批改展示分数与评语。
  - 若未组队：引导「创建小组」或「加入小组」；组队入口可复用课程详情内组队大厅逻辑（按 section），或在本页内按当前学生所在 section 展示可加入的小组列表与创建小组入口。组队完成后刷新「我的小组」与提交区。
- **个人任务**（当任务 allow_group = false 时）：不显示组队区；直接显示提交区：未提交则显示提交表单（内容 + 可选链接）；已提交则显示提交记录与时间；已批改则显示分数、评语。
- **提交区**：
  - 已组队且未提交：提交表单（content 必填，attachment_url 本阶段可不做）；提交时：`submission.insert({ task_id, group_id, content })`（不填 student_id）。
  - 个人任务未提交：提交表单；提交时：`submission.insert({ task_id, student_id, content })`（不填 group_id）。
  - 已提交：显示 content、submitted_at；若 status = 'graded' 显示 score、feedback。
- **权限**：校验当前学生已选该 course（通过 enrollment → section → course_id）；且若组队任务，只能为自己所在 section 的组提交。

### 4.7 与现有组队逻辑的衔接

- 学生组队按 **section** 维度，已在课程详情「组队大厅」完成；任务详情页若需组队，直接复用「当前学生在某 section 下是否已有 group」的查询结果；提交时用该 group_id。若学生在多 section 下选同一课程，需约定「以某一 section 为准」（如第一个 section）用于该任务的组与提交，或按产品约定多 section 分别显示（本阶段可简化为：取当前学生在该 course 下第一个 section 的 group）。
- 教师端「应提交总数」：组队任务 = 该 course 下所有 section 的 group 数量之和；个人任务 = 该 course 下 enrollment 去重 student 数。实现时用 Supabase 查询或前端汇总均可。

## 六、与现有代码的衔接

- **Supabase 客户端**：全站统一使用 `lib/supabase.ts` 的 `createClient()`。
- **Auth**：教师端用 `user.role === 'teacher'` 且 `user.profile_id === course.teacher_id`；学生端用 `user.profile_id` 即 student_id，enrollment 校验选课与 section。
- **样式**：沿用设计基线（`rounded-card`、`shadow-card`、`primary` 按钮、表格与卡片），与阶段 4/5/6 一致。
- **路由**：教师/学生 layout 与课程详情页已存在；新增 `tasks/new`、`tasks/[taskId]`、`tasks/[taskId]/grade/[submissionId]` 子路由；课程详情内任务区块由占位改为真实数据与链接。
- **组队**：学生端任务详情若需组队，复用现有「按 section 查 group、group_member、我的 group」逻辑；可调用同一套 refreshGroups 或在本页内单独查当前学生在当前 course 下某 section 的 group。

## 七、验收标准

- 教师在 Supabase 执行建表 SQL 后，可在某课程下发布任务（组队/个人均可），在课程详情任务列表中看到该任务，进入任务详情看到提交列表（初始为空）。
- 学生在该课程下看到任务列表，进入任务详情可完成组队（若需组队）并提交；提交后教师端任务详情出现该条提交，教师可进入批改页打分与评语并发布成绩。
- 学生端任务详情在教师发布成绩后能展示分数与评语。
- 空态、加载态、错误态（如 task 不存在、无权限）有合理提示，不崩溃。
- 每人/每组每任务只能提交一次；批改后状态为已批改，学生可见成绩。
