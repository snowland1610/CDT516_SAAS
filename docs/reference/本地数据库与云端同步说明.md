# 本地数据库与 Supabase 云端同步说明

> 目标：本地先管理数据库、完成功能开发与演示，最后同步到 Supabase，使部署后其它设备和用户可访问。

---

## 一、总体思路

- **开发阶段**：在本地跑一个与 Supabase 兼容的数据库（推荐 PostgreSQL），用同一套建表与 seed SQL，前端连接本地库开发、联调。
- **上线/演示阶段**：把「表结构 + 可选的数据」同步到 Supabase 云端，前端改为连接 Supabase（或通过环境变量切换），部署后所有人访问同一套云端数据。

只要本地和云端都用 **PostgreSQL**，且表结构与 Supabase 一致，迁移时只需「在 Supabase 上执行同样的 SQL」或「用 Supabase CLI 推送迁移」即可，无需换库或大改代码。

---

## 二、两种常见做法

### 方案 A：Supabase 本地开发（推荐）

用 **Supabase CLI** 在本地启动一整套 Supabase（含 PostgreSQL、与云端同版本），表结构用 **迁移文件** 管理，数据可本地 seed；要上云时把迁移推到远程项目即可。

| 优点 | 说明 |
|------|------|
| 与云端一致 | 本地和 Supabase 都是同一套引擎与扩展，RLS、Auth 等可本地测 |
| 迁移省事 | 表结构在 `supabase/migrations/*.sql` 里，用 `supabase db push` 即可同步到云端 |
| 官方支持 | Supabase 文档推荐此方式做本地开发 |

| 你需要做的 | 说明 |
|------------|------|
| 安装 Supabase CLI | 见 [Supabase CLI](https://supabase.com/docs/guides/cli) |
| 在项目里初始化 | `supabase init`（若已有 `supabase/` 可保留），把现有建表 + seed 拆成 migration 和 seed 文件 |
| 本地启动 | `supabase start`，会得到本地 DB URL、anon key 等，前端用这些连本地 |
| 上云 | `supabase link --project-ref <你的项目ref>` 后 `supabase db push`，再在 Supabase 网页执行 seed（或单独 seed 脚本跑一次） |

前端通过 **环境变量** 区分：本地用 `.env.local` 指向本地 Supabase URL/Key，部署时用 Vercel/Netlify 等配置 Supabase 云端 URL/Key，代码不用改。

---

### 方案 B：本地单独跑 PostgreSQL

不装 Supabase CLI，只在本地跑一个 **PostgreSQL**（本机安装或 Docker），用你现有的 `schema-and-seed.sql` 建表并插入演示数据；前端连接本地 PostgreSQL（需用 `@supabase/supabase-js` 时，Supabase 客户端要连 Supabase 的 REST/API，不能直连任意 PostgreSQL，见下）。

注意：  
- 若前端用的是 **Supabase Client**（`createClient(url, key)`），请求走的是 Supabase 的 **REST API**，不是直连 PostgreSQL。所以「本地 PostgreSQL」不能直接当 Supabase 用，除非你本地跑的是 **Supabase 整套服务**（即方案 A）。  
- 若你希望「本地一个纯 PostgreSQL」，前端就要改成直连数据库（例如用 Prisma、Drizzle、或 Node 的 pg），这样和 Supabase 的用法不一致，之后上云要切回 Supabase Client 并保证表结构一致。

因此：  
- **若坚持用 Supabase Client 做前端**：适合用 **方案 A（Supabase 本地）**，这样本地和云端都是「Supabase API + PostgreSQL」，只是 URL/Key 不同。  
- **若接受前端直连 PostgreSQL**：可用方案 B 在本地开发，表结构用同一份 SQL；上云时在 Supabase SQL Editor 执行同一份 SQL，再把前端连接改为 Supabase（或加一层后端连 Supabase）。

---

## 三、推荐结论

- **若你打算全程用 Supabase 做后端（前端用 Supabase Client）**：  
  → 用 **方案 A**：Supabase 本地开发，表结构用 migrations，本地 `supabase start` 开发，最后 `supabase db push` + 在云端执行一次 seed（或脚本），部署时前端切到云端 URL/Key。
- **若你暂时只想「本地一个库、能跑 SQL 即可」**：  
  → 可以本地跑 PostgreSQL（如 Docker），用现有 `schema-and-seed.sql` 建表与 seed，但前端若用 Supabase Client，仍需在本地跑 Supabase（方案 A），否则只能当「纯本地数据库 + 其他方式访问」，不能和当前 Supabase 前端方案无缝衔接。

---

## 四、按方案 A 的简要步骤（本地先管理，再上云）

1. **安装并登录 Supabase CLI**  
   - 安装：见官网；登录：`supabase login`。

2. **在项目里准备好迁移与 seed**  
   - 把现有建表 SQL 放到 `supabase/migrations/`（例如 `20250101000000_initial.sql`），只放 `CREATE TABLE` 等 DDL。  
   - 把插入演示数据的 SQL 放到 `supabase/seed.sql`（或用你现有的 seed 内容）。

3. **本地启动**  
   - `supabase start`。  
   - 用终端输出的 `DB URL`、`anon key` 等填到前端 `.env.local`，前端连接本地 Supabase 完成功能开发。

4. **本地开发与调试**  
   - 表结构变更时，新增 migration 文件（或改现有 migration 后重置本地 DB：`supabase db reset`）。  
   - 需要时改 `seed.sql` 并执行 `supabase db reset` 重新灌数据。

5. **同步到 Supabase 云端**  
   - `supabase link --project-ref <项目 ref>`（项目 ref 在 Supabase 项目设置里）。  
   - `supabase db push`：把 migrations 推到云端，在云端建好表。  
   - 若云端是空库，在 Supabase SQL Editor 里执行 `seed.sql`，或配置 `supabase/seed.sql` 在 push 后自动执行（视 CLI 版本而定），这样云端也有演示数据。

6. **部署前端**  
   - 部署时环境变量使用 Supabase 云端的 **Project URL** 和 **anon key**，用户与其它设备访问的即是云端数据。

这样，你就能在本地先管理并开发完我们讨论的所有功能，最后通过「推送迁移 + 执行 seed」上传到 Supabase，实现部署后其它设备和用户共同操作同一套云端数据库。
